## 単語埋め込み

- LLMをはじめとするDLモデルはRaw text直接は処理できない設計のため、連続数値であるベクトルとして各txtを表現することがよく行われる
  - 数値以外のデータをNNが処理できる形式へ変換する必要がある
  - TODO: もう少し上手く言語化する(vectorにする理由は扱いやすくするため..というのを)
- データをベクトルのフォーマットへ変換するという概念が**埋め込み(embedding)**と呼ばれる
  - 埋め込みは特定のNNモデルを使って行われる。一般に、**audioにはaudio用の、textにはtext用の埋め込みモデルが必要**
  - 言い換えると埋め込み後のベクトルは、文書内の単語や文書全体、画像などの離散値のオブジェクトが連続値のベクトル空間へ写像されたもの、といえる
- 以降では、**単語の埋め込み**に焦点を当てる
  - 段落や文書全体埋め込みは**RAG(Retrieval-Augmented Generation)**でよく使われる
- 単語埋め込みを生成するためのアルゴリズムやフレームワークはいくつかある
  - **word2vec**
    - NNアーキテクチャを訓練し、目的の単語からその周辺のコンテキストを予測するか、コンテキストの単語群から目的の単語を予測することで単語埋め込みを生成する
    - 「似たようなコンテキストに現れる単語は、似たような意味を持つ傾向がある」という考え方がベース
      - のため、二次元の単語埋め込みを可視化すると、類似する意味を持つ単語によってクラスタが生成される
- **埋め込みの次元数**
  - 次元が高くなるほど、より細かい関係を捉えることが可能になる一方、計算効率が犠牲になる
- LLMは埋め込みを独自に生成する
  - "隠れ状態の次元数"とも呼ばれるそう
  - 例えば、GPT-3(1.75B)は埋め込みサイズは12,288次元

### context token

- `<|endoftext|>`(テキストソースを分割)や`<|unk|>`(未知の単語)などの特別なcontextを持つtokenを使用することで、「連結されたtxt dataだけど、これらは無関係」ということを表すことができるので、モデルの理解を向上させることができる
  - 追加するtokenは調整の余地があって、`[BOS]`や`[EOS]`などもこれにあたる
- `./simple_tokenizer.py`参照
